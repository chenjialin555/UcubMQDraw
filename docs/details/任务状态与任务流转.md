# 任务状态与任务流转

> 状态枚举的 API 权威定义见 [API接口.md](../API接口.md#十一任务状态枚举)。

## 一、状态定义

| 状态 | 含义 | 前端展示 |
|------|------|----------|
| `created` | 任务已创建，尚未入队 | 已创建 |
| `queued` | 已入队，等待 imggen 消费 | 排队中 |
| `dispatching` | 正在投递 MQ | 提交中 |
| `running` | imggen / ComfyUI 正在执行 | 生成中 |
| `succeeded` | 执行成功（终态） | 已完成 |
| `failed` | 执行失败（终态） | 生成失败 |
| `canceled` | 用户取消（终态） | 已取消 |

**统一使用 `succeeded`，不要混用 `completed` / `success`。**

imggen MQ 回调字段仍为 `status: "success"`（见 [mq_contract.md](../mq_contract.md)），后端映射为 `succeeded`。

---

## 二、状态流转

```text
created → dispatching → queued → running → succeeded
                                      ├→ failed
                                      └→ canceled（用户取消）
```

创建任务正式链路：

```text
1. 写 DB，status=created
2. WS task.created
3. status=dispatching
4. 投递 RocketMQ
5. 成功 → queued；失败 → failed
6. WS task.updated / task.failed
```

Mock 模式按上述顺序模拟，每步间隔约 1 秒，终态为 `succeeded`。

---

## 三、终态保护

终态：`succeeded` / `failed` / `canceled`。

```python
if task.status in ("succeeded", "failed", "canceled"):
    return task  # 不允许迟到的 running 回调覆盖
```

重复 success 回调：可 merge `task_callback_params` raw，但不重复推错误状态。

## 四、前端「进行中」筛选

```text
created / queued / dispatching / running
```

```ts
const RUNNING_STATUSES = ['created', 'queued', 'dispatching', 'running']
```

右栏筛选 Tab 建议：

| Tab | 规则 |
|-----|------|
| 全部 | 所有未删除任务 |
| 进行中 | `RUNNING_STATUSES` |
| 已完成 | `succeeded` |
| 失败 | `failed` |
| 收藏 | `favorite === true` |

## 五、完整任务链路（正式版）

```text
用户上传图 → OSS URL 写入表单
  ↓
tool.buildRequest() → POST /tasks（HTTP）
  ↓
后端 validate → 写 DB（user_id + 三列 JSON）
  ↓
RocketMQ 投递（或 Mock 线程）
  ↓
imggen 执行
  ↓
MQ callback → 幂等更新 DB
  ↓
WebSocket send_to_user(task.user_id)
  ↓
taskStore.upsertTask → 右栏更新
```

**业务命令（删除 / 再次生成 / 取消 / 收藏）全部走 HTTP，不走 WebSocket。**

## 七、回调后如何更新任务

1. `handle_imggen_callback` 根据 `innerTaskId` 查任务
2. 终态保护检查
3. 写入 `task_callback_params`、更新 `status` / `progress` / `cost_time` / `finished_at`
4. `websocket_manager.send_to_user(task.user_id, event)`
5. 前端 `upsertTask` 合并；断线期间靠 `fetchTasks` 补齐

详见 [WebSocket推送机制.md](./WebSocket推送机制.md)、[RocketMQ对接设计.md](./RocketMQ对接设计.md)。

## 八、任务操作（HTTP）

| 操作 | 接口 | 说明 |
|------|------|------|
| 再次生成 | `POST .../tasks/{id}/rerun` | 复制 `taskSubmitParams`，**新建** taskId |
| 重新编辑 | 无专用接口 | 前端读 `taskSubmitParams` 回填表单 → POST 创建新任务 |
| 删除 | `DELETE .../tasks/{id}` | 软删除 + WS `task.deleted` |
| 取消 | `POST .../tasks/{id}/cancel` | 进行中可取消 → `canceled` |
| 收藏 | `PATCH .../tasks/{id}/favorite` | `{ "favorite": true }` |

## 九、开发 Mock 与正式版差异

| 项 | `USE_MOCK=true` | 正式版 |
|----|-----------------|--------|
| MQ | 跳过 | RocketMQ 投递 + callback |
| WS 推送 | 仍建议 `send_to_user` | 必须按 user_id 定向 |
| 用户隔离 | 可暂用固定 user_id | token 解析 |
| 分页 | 可暂返回全量 | 必须分页 |
