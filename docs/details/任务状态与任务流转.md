# 任务状态与任务流转

> 状态枚举的 API 权威定义见 [API接口.md](../API接口.md#七任务状态枚举)。

## 一、状态定义

| 状态 | 含义 | 前端展示 |
|------|------|----------|
| `created` | 任务已创建，尚未入队 | 一般不单独展示 |
| `queued` | 已入队，等待调度 | 排队中 |
| `dispatching` | 正在投递 MQ / 分配算力 | 准备调度 |
| `running` | imggen 正在执行 | 进度条 |
| `completed` | 执行成功 | 展示输出图 |
| `failed` | 执行失败 | 展示 `errorMessage` |

第一版 MVP **不实现**：`cancelled`、`retrying`。

## 二、状态流转

```text
created → queued → dispatching → running → completed
                                      └→ failed
```

Mock 模式会按上述顺序模拟，每步间隔约 1 秒。

## 三、前端「进行中」筛选

以下状态视为进行中，右栏显示进度条：

```text
created / queued / dispatching / running
```

```ts
const RUNNING_STATUSES = ['created', 'queued', 'dispatching', 'running']
```

## 四、完整任务链路

```text
用户上传图 → url 写入表单
  ↓
tool.buildRequest() → POST /tasks
  ↓
后端 validate → 写 DB（task_submit_params + task_dispatch_params）
  ↓
Mock 线程 / MQ 发送
  ↓
imggen 执行
  ↓
MQ 回调 → 更新 task_callback_params
  ↓
WebSocket task_update
  ↓
taskStore.upsertTask → 右栏更新
```

## 五、TaskCard 预览图解析

不单独存 `input_preview` / `output_preview`，从 JSON 按需解析：

```ts
// 输入：有 refImageList 取 [0]；文生图无图则占位
task.taskSubmitParams?.refImageList?.[0]

// 输出：回调里的 imageUrlList
task.taskCallbackParams?.imageUrlList?.[0]
```

## 六、回调后如何更新任务

1. `handle_imggen_callback` 根据 `innerTaskId` 查任务
2. 写入 `task_callback_params`、更新 `status` / `progress` / `cost_time`
3. `websocket_manager.broadcast_task_update(task)`
4. 前端 `upsertTask` 合并到列表

详见 [WebSocket推送机制.md](./WebSocket推送机制.md)、[RocketMQ对接设计.md](./RocketMQ对接设计.md)。

## 七、MVP 不做的能力

| 能力 | 原因 |
|------|------|
| 取消任务 | 需 imggen 侧支持，第一版砍掉 |
| 重跑 | 第二版基于 `task_submit_params` 创建新任务 |
| 收藏 | 非主链路，表字段 `favorite` 预留即可 |
